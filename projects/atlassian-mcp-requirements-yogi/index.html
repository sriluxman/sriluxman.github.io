Project / building

Atlassian MCP + Requirements Yogi

Extending an Atlassian MCP server so agents can reason about requirements with clearer structure and provenance.

Requirements are stored as pages and tickets, but they are used as an evolving argument: this constraint exists because of that customer, this decision supersedes that one, this claim is still contested.

Tooling mostly ignores the argument and serves the storage. When agents arrive, that gap becomes expensive — an agent reading a requirements space without structure and provenance will confidently synthesize text from stale, contradictory, or superseded sources.

What it is

This project extends an Atlassian MCP server around Requirements Yogi so that agents can navigate requirements as a connected surface rather than a pile of pages: finding the evidence behind a requirement, keeping relationships between requirements visible, and preserving where each claim came from.

The working idea is simple: requirements tools should help explain why the system is becoming what it is. If an agent can answer that, it can meaningfully assist with impact analysis, consistency checking, and the drudge work of keeping requirement sets coherent. If it cannot, it is autocomplete wearing a requirements badge.

Why I care

I have spent years on the product side of requirements engineering, watching structure decay the moment delivery pressure rises. Making requirement reasoning cheap enough that agents can hold the structure for us is one of the most practical, least glamorous applications of agentic engineering I know — which is usually the sign of a good one.
